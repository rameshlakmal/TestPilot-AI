AI Test Case Generator — Workflow

Complete internal workflow documentation showing how the system processes requirements into structured test scenarios across all 3 user paths.

Architecture Overview

The platform is a two-tier application: an Express.js backend (server/) that orchestrates LLM calls and file processing, and a React + Material UI frontend (client/) that provides the user interface. Static files for production are served from web/. The backend communicates with one of three AI providers (OpenAI, Anthropic, or Gemini) to analyze requirements and generate test cases. Twelve markdown-based "skill" files in the skills/ directory teach the AI how to apply specific testing techniques.

How skills work: Each skill is a markdown file in skills/ that teaches the AI a specific test design technique (e.g., Boundary Value Analysis, State Transition Testing). The file includes guidance on when to apply the technique, what to look for, and example patterns. These are injected into LLM prompts so the AI applies formal QA methods rather than guessing.

Every path starts the same way: the user provides a requirement. This can be plain text typed/pasted into a textarea, a file upload, or imported directly from Jira user stories. The server processes uploaded files into plain text before anything else happens.

Depth Levels

Before triggering any path, the user selects a depth level that controls how many test cases the AI should generate. This affects the prompt instructions and the min/max constraints applied to the output.

Path A — Ask Missing Info First (Detailed)

Path A is a pre-processing step that helps when requirements are vague or incomplete. It doesn't generate test cases directly — instead, it identifies what's unclear and asks the user to fill in the gaps. After answering, the user proceeds to Path B (Analyze & Recommend) with a richer, more complete requirement.

User clicks "Ask missing info first"

Requirement text + optional file sent as multipart form data to the server

▼

POST /api/preflight

The server processes the request in several steps:

  1. Parse the uploaded file (if any) using the file extraction pipeline described above
  2. Combine extracted file text with the typed requirement text
  3. Run keyword-based skill selection to identify which QA techniques are relevant (see Skill Selection below)
  4. Load the markdown content of matched skills
▼

Build the Preflight Prompt

Two prompts are constructed and sent to the LLM:

  • System prompt: "You are a senior QA engineer and business analyst. Identify ambiguity and missing information. Return strict JSON only."
  • User prompt: Contains the full requirement text + the markdown content of selected skills as context, plus the expected output schema

The AI is instructed to keep questions concise and actionable, avoid redundant questions, and focus only on information that would materially affect test case design.

▼

LLM Call — Preflight Analysis

Single LLM call at temperature 0.2 with a 90-second timeout. The AI analyzes the requirement and returns structured JSON.

▼

Validate Response (AJV)

The raw LLM output goes through the schema validation pipeline:

  1. Try JSON.parse() directly on the response text
  2. If that fails, scan for the first { and extract the JSON object
  3. Validate the parsed object against the preflight schema using AJV
  4. If validation fails, send a repair prompt back to the LLM (temperature lowered to 0.1) with the validation errors, asking it to fix and return corrected JSON
▼

The user's answers are formatted as "Q: [question] A: [answer]" pairs and appended to the original requirement text. This enriched requirement is then sent to the Analyze endpoint, giving the AI much better context to work with.

Path B — Analyze & Generate Per-Skill (Detailed)

This is the recommended workflow for highest quality output. It works in two phases: first the AI analyzes the requirement to understand what needs testing and which techniques to use, then it generates test cases for each technique in separate, focused LLM calls running in parallel.

Phase 1 — Requirement Analysis

The analysis phase extracts testable elements from the requirement and recommends which of the 12 available testing techniques would be most effective. This gives the user visibility and control before generation begins.

User clicks "Analyze & recommend techniques"

Requirement text + optional file sent to the server. No skill pre-selection needed — the AI figures out which techniques apply.

▼

POST /api/analyze

The server builds an analysis prompt:

  • System prompt: "You are a senior QA engineer and test architect. Analyze the requirement and recommend applicable test techniques."
  • User prompt: Contains the requirement text + a list of all 12 available skills (just their ID, title, and tags — not the full markdown) + the expected output schema

Key instructions to the AI: extract testable elements of different types, recommend only 3-6 techniques that genuinely apply (not all of them), assign confidence levels, keep descriptions to 1-2 sentences max.

▼

LLM Call — Analyze Requirement

Single LLM call at temperature 0.2. The AI reads the requirement holistically and returns a structured analysis.

▼

  • Always inject "general-fallback" skill if it's not already in the recommendations — this ensures baseline coverage regardless of what the AI recommends

User Reviews Recommendations in the UI

The UI displays the extracted elements as a summary, then shows each recommended technique as a card with a checkbox. High and medium confidence techniques are pre-checked. Low confidence techniques are unchecked but visible. The user can toggle any technique on or off before proceeding.

▼

Optional: Select Technique Diagrams

Below the technique list, an "Available Diagrams" section appears showing which visual diagrams can be generated for the selected techniques. Diagrams are entirely optional — none are selected by default.

Each selected technique maps to a Mermaid.js diagram type:

  • State Transition Testing → stateDiagram-v2 (states, transitions, invalid transitions)
  • Decision Tables → flowchart TD (conditions branching to outcomes)
  • Equivalence Partitioning → flowchart LR (valid/invalid input partition blocks)
  • Boundary Value Analysis → flowchart LR (boundary points with pass/fail zones)
  • Pairwise / Combinatorial → flowchart TD (parameter combination tree/matrix)
  • Feature Decomposition → mindmap (actors, data, rules, states, integrations)
  • Other techniques → flowchart TD (logical flow of the technique)

The user can check individual diagrams or use "Select all / Deselect all". Only checked diagrams are requested from the LLM — this avoids wasting tokens when diagrams aren't needed.

Phase 2 — Per-Skill Parallel Generation

This is the core innovation of Path B. Instead of dumping all techniques into one prompt, each selected technique gets its own dedicated, focused LLM call. This means the AI can deeply apply one technique at a time, producing higher quality and more thorough test cases. The calls run in parallel for speed.

User clicks "Generate test scenarios"

The selected skill IDs + optional diagram skill IDs + requirement text + analysis context (extracted elements) are sent to the server. The diagramSkills parameter tells the server which skills should also produce a Mermaid diagram.

▼

POST /api/generate-tests (per-skill mode)

The server detects selectedSkills in the request body and enters the per-skill generation path. For each selected skill, it creates a separate generation task:

  • System prompt: "You are a senior QA engineer specializing in [technique name]. Generate test cases using THIS technique only."
  • User prompt: Contains the requirement + only that skill's markdown (not all 12) + the extracted elements from Phase 1 as additional context + depth-based instructions
  • The AI is told to focus only on scenarios this specific technique uncovers and not duplicate generic happy-path testing unless it's technique-specific
  • Conditional diagram: If the user selected a diagram for this skill, the system prompt also instructs the AI to include a mermaidDiagram field containing a valid Mermaid.js diagram string. The schema hint also includes the diagram field only when requested. Skills without diagram selection skip this entirely — no extra tokens are consumed.
▼

Parallel LLM Calls

All skill generation tasks are executed in parallel with a concurrency limiter: up to 3 simultaneous calls for OpenAI and Anthropic, but only 1 at a time for Gemini (due to free-tier rate limits). Each call operates independently — if one skill's generation fails, it doesn't crash the others.

LLM Call #1
Boundary Value Analysis prompt → focused on min/max edges, off-by-one errors → 5-10 boundary scenarios
LLM Call #2
State Transition prompt → focused on workflow states, valid/invalid transitions → 5-10 state scenarios
LLM Call #3
Error Guessing prompt → focused on common failure patterns, edge cases → 5-10 error scenarios
LLM Call #4
General Fallback prompt → baseline happy path + core coverage → 5-10 baseline scenarios

Each call's output is independently validated against the test suite schema. If the output is invalid, a repair prompt is sent (temperature lowered to 0.1) with up to 12 sample validation errors. If the repair also fails, that skill returns an empty result — the other skills' results are still used. Any skill-level errors are reported in the skillErrors field of the response.

▼

Extract Diagrams & Merge All Suites

Before merging, the server extracts any mermaidDiagram fields from each skill's suite and collects them into a separate skillDiagrams array (each entry contains skillId, skillTitle, and diagram string). This must happen before merging because the merge process constructs a new combined object.

Then mergeSuites() combines the test case results from all skill calls:

  • Suite title becomes "Skill1 + Skill2 + Skill3 + ..." (or "Generated Test Suite" as fallback)
  • Assumptions, risks, and missing info questions are merged and deduplicated (case-insensitive)
  • All test cases from all skills are combined into a single array
  • Each test case retains a coverageTags field identifying which skill generated it
▼

Jaccard Deduplication

When multiple skills generate test cases independently, there's often overlap. For example, both "Boundary Value Analysis" and "Equivalence Partitioning" might generate a test for "empty input field". The system uses weighted Jaccard similarity to detect and remove near-duplicates — not just exact matches.

How it works:

  1. Tokenize: Each test case's title, steps, and expected results are split into lowercase tokens (punctuation removed, words under 2 characters filtered out)
  2. Compare: For each pair of test cases, compute Jaccard similarity (intersection / union of token sets) for title, steps, and expected results separately
  3. Weight: Combine the three similarity scores using fixed weights
  4. Threshold: If the weighted score is ≥ 60%, the test case is marked as a duplicate and removed

The algorithm iterates through test cases in order. For each case, it checks against all previously "kept" cases. If any comparison exceeds the threshold, the newer case is discarded. Duplicate groups (which cases were removed and what they were similar to, with percentage scores) are returned to the UI so the user can see what was filtered out.

▼

Renumber IDs

After dedup and merging, all remaining test cases are renumbered sequentially: TC-001, TC-002, TC-003, ... This ensures clean, gap-free IDs regardless of how many cases were removed.

Final Test Suite Returned

Each test case is tagged with the skill that generated it via coverageTags, giving full traceability back to the technique used.

Technique Diagrams (Mermaid.js)

An optional feature that generates visual diagrams showing how each test design technique applies to the given requirement. Diagrams are generated by the LLM alongside test cases and rendered client-side using Mermaid.js as interactive SVGs.

How It Works

1. User Selects Diagrams (Analyze Step)

After the AI recommends techniques, an "Available Diagrams" panel shows which techniques support visual diagrams. The user checks only the diagrams they want. None are selected by default — this is a zero-cost feature when not used.

▼

2. Conditional Prompt Injection (Server)

The server receives diagramSkills (array of skill IDs). For each skill in the list, the system prompt gets additional instructions to include a mermaidDiagram field with a valid Mermaid.js diagram string. Skills NOT in the list get no diagram instructions — saving tokens and keeping the prompt focused on test cases.

▼

3. LLM Returns Diagram + Test Cases

The LLM returns a JSON object that includes both testCases and mermaidDiagram. The diagram is a string containing valid Mermaid syntax (e.g. stateDiagram-v2, flowchart TD, mindmap). The AJV schema allows mermaidDiagram as an optional string field.

▼

4. Diagram Extraction (Before Merge)

Before suites are merged, the server extracts each skill's mermaidDiagram into a separate skillDiagrams array: [{ skillId, skillTitle, diagram }]. This is necessary because mergeSuites() constructs a new object and would lose the diagrams.

▼

5. Client Renders with Mermaid.js

The MermaidDiagram React component initializes Mermaid.js with a dark theme (matching the app's purple accent), strips any accidental code fences, and calls mermaid.render() to produce an inline SVG. Diagrams appear as collapsible accordion panels in the Results step under "Technique Diagrams". Render errors are caught gracefully and shown as a small error message instead of crashing the UI.

Test Case Output Structure

Both paths produce test suites in the same JSON format. Here's the exact structure that the LLM is asked to return, and that the AJV schema validates against:

{ "suiteTitle": "Login Feature Test Suite", "assumptions": ["User has a registered account", "..."], "risks": ["Third-party auth provider downtime", "..."], "missingInfoQuestions": ["Is MFA required for all users?", "..."], "testCases": [ { "id": "TC-001", "title": "Verify successful login with valid credentials", "type": "functional", // functional | negative | boundary | security | accessibility // | performance | usability | compatibility | resilience "priority": "P0", // P0 (critical) | P1 (high) | P2 (medium) | P3 (low) "preconditions": ["User account exists", "User is on login page"], "steps": [ // min 1 step required "Enter valid username", "Enter valid password", "Click Login button" ], "expected": [ // min 1 expected result required "User is redirected to dashboard", "Welcome message displays username" ], "coverageTags": ["functional-core", "happy-path"], "requirementRefs": ["REQ-LOGIN-001"] } ] }

Schema Validation & Auto-Repair Pipeline

LLMs don't always return perfect JSON. They might wrap output in markdown code fences, include commentary before/after the JSON, produce malformed syntax, or miss required fields. Every LLM response in the system goes through a multi-step validation pipeline to ensure the output is valid, structured, and safe to use.

LLM Response (raw text)

May contain valid JSON, markdown-wrapped JSON, commentary mixed with JSON, or completely malformed output

▼

Step 1 — Direct JSON Parse

Try JSON.parse() on the raw response text. If it succeeds, proceed directly to schema validation. This handles the happy path where the LLM returns clean JSON.

▼

Step 2 — Extract JSON from Text

If direct parse fails, scan the text for the first { or [ character and attempt to extract a complete JSON object/array from that point. This handles cases where the LLM wraps JSON in markdown code fences (```json ... ```) or adds explanation text before/after. The extraction handles nested braces, string escapes, and balanced brackets correctly.

▼

Step 3 — AJV Schema Validation

The parsed JSON object is validated against the expected schema (analysis, testSuite, or preflight) using the AJV JSON Schema validator. This checks required fields, enum values, array constraints (e.g., steps must have at least 1 item), string formats, and additional properties.

▼

Step 4 — Auto-Repair (if validation fails)

If the parsed JSON doesn't match the schema, the system sends a repair prompt back to the LLM:

  • Temperature is lowered to 0.1 (more deterministic) for the repair call
  • The prompt includes up to 12 sample validation errors from AJV, telling the AI exactly what's wrong
  • Message: "Your output was invalid. Here are the errors: [...]. Fix and return only the corrected JSON."
  • The repaired output goes through the same parse → validate pipeline again
  • If the repair also fails, the system returns an error for that specific call (in per-skill mode, other skills still succeed)
▼

Step 5 — Post-Processing & Normalization

After validation passes, the output is normalized:

  • All string arrays are trimmed (whitespace removed from start/end of each string)
  • Empty strings are filtered out
  • Arrays are capped at 500 items, individual strings at 500 characters
  • Test case count is enforced against the depth-based maximum
  • IDs are renumbered sequentially (TC-001, TC-002, ...)
▼

Valid Structured Output

Type-safe, schema-compliant JSON ready for the UI. The repaired field in the API response indicates whether a repair prompt was needed.

LLM Provider Details

The system supports three AI providers. Each has its own adapter in server/llm/ that handles authentication, request formatting, response parsing, and error recovery differently. All providers use temperature 0.2 by default and a 90-second timeout.

ProviderDefault ModelJSON ModeSpecial Handling
OpenAI gpt-4.1-mini Structured output via response_format: {type: "json_schema"} with strict mode. Falls back to {type: "json_object"} if the model rejects the schema. AbortController for timeout. System message goes in the messages array.
Anthropic claude-3-5-sonnet-latest No native JSON schema enforcement — relies entirely on prompt guidance to produce valid JSON. Uses anthropic-version: 2023-06-01 header. System prompt is a separate top-level field, not part of the messages array. Default max_tokens: 2000.
Gemini gemini-1.5-flash Uses responseMimeType: "application/json". Optionally sends responseSchema and function declarations, with multiple fallback layers if the model rejects them. Rate limit retry: On 429 errors, parses "retry in Xs" from the error message, waits, and retries up to 3 times. Concurrency limited to 1 parallel call.

AIO / Jira Integration

After generating test cases, users can push them directly to AIO Test Management for Jira. The integration creates test cases in the specified Jira project with proper folder hierarchy, priorities, steps, and tags.

User Configures Push Settings

Enters Jira Project ID (required), folder path (optional, e.g. "Automated/Sprint 12"), AIO token (or uses env var), and whether to include coverage tags.

▼

POST /api/aio/push

The server processes the push in several steps:

  1. Create folder hierarchy — Splits the folder path by /, >, or \ and creates the folder tree in AIO via PUT /api/v1/project/{id}/testcase/folder/hierarchy
  2. Fetch priority config — Gets the project's priority list from AIO and maps P0→Critical/Highest/Blocker, P1→High, P2→Medium/Normal, P3→Low/Lowest/Minor
  3. Fetch script type — Gets the default test case script type (prefers "classic" if available)
  4. Create tags — If coverage tags are enabled, bulk-creates tags from each test case's coverageTags array
  5. Create test cases — For each test case, builds an AIO-compatible payload with title, description (preconditions), steps with expected results, priority, folder, and tags
▼

Step Mapping Logic

Test case steps are mapped to AIO's step format with intelligent pairing:

  • If no steps defined: creates a single step from the test case title with all expected results
  • If steps.length === expected.length: pairs them 1:1 (step 1 → expected 1, step 2 → expected 2, ...)
  • If only 1 expected result: applies it to the last step only
  • Otherwise: joins all expected results and applies them to the last step
▼

Push Result

Returns the count of created test cases and their AIO case IDs. The UI displays a log of what was created.

The React UI persists user preferences to localStorage under the key ai-test-generator:v1. This means your settings survive page refreshes and browser restarts. The following fields are saved:

Provider
Selected AI provider (openai / anthropic / gemini)
Depth
Selected depth level (smoke / standard / deep)
Model Override
Custom model name if the user overrode the default
Requirement Text
The last entered requirement text (auto-saved as you type)
Preflight Answers
User's answers to clarifying questions (question → answer map)
AIO Settings
Jira project ID, folder path, and coverage tag preference
